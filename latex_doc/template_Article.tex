\section{引言}
\subsection{编写目的}
本文档用于系统阐述【版权云链】在软件版权溯源与确权场景下的总体架构、模块设计、接口规范、数据结构与安全机制，为系统的研发实现、测试验收、运行维护和后续迭代提供统一技术依据。与此同时，文档作为项目建设的重要技术佐证，明确系统如何落实国家在知识产权保护、数字经济高质量发展与网络信息安全治理方面的政策导向，确保平台在“可用、可信、可审计、可持续”四个维度具备工程落地能力。

\subsection{项目背景}
在日常软件开发和过往软件著作权申请实践中，遇到过软件成果被第三方随意复制、传播和使用的情况。更突出的问题是，部分第三方仅对原有软件进行轻度修改（如界面文字替换、局部代码调整、配置参数变更）后，即尝试以“新作品”名义申请软著，导致原创方在权属证明、侵权识别和争议举证环节面临较高成本与较长周期。这类现象反映出传统“静态材料提交 + 事后争议处理”模式在软件资产快速迭代场景下存在明显不足，尤其缺乏可追溯、可比对、可验证的持续性证据链。

从国家层面看，知识产权强国建设持续推进，数字内容确权、版权保护和侵权治理能力建设成为数字经济高质量发展的关键基础；同时，《中华人民共和国网络安全法》《中华人民共和国数据安全法》《中华人民共和国个人信息保护法》等法律法规对数据处理安全、系统安全防护、日志留痕与责任可追溯提出了更高要求。在此背景下，软件版权保护不仅是单一业务需求，更是企业合规治理、创新成果保护和数字资产管理能力的重要组成部分。

基于上述痛点与政策要求，项目拟建设“版权云链”软件版权溯源与确权平台，探索覆盖“提交存证、指纹提取、相似性识别、风险分级、人工复核、链上锚定、公开核验”的全流程技术方案。平台通过“链下保存原始文件、链上锚定关键摘要”的方式，将软件版本、提交主体、处理过程和确权结果形成可回放、可审计的证据闭环，提升软件版权保护的及时性与可信度，降低原创方维权取证成本，并为后续版权治理、合规审计和行业规范化提供可复用的技术基础。


\section{项目技术选型与运行环境}
\subsection{关键技术与框架}

\subsubsection{Vue 3 + Vite + Element Plus}
系统前端采用 Vue 3 作为核心视图框架，使用组合式 API 构建可复用业务组件；Vite 作为构建工具，提供快速热更新与高效打包能力，提升研发迭代效率；Element Plus 作为 UI 组件库，用于统一页面风格、表单交互和后台管理界面规范，保障系统在功能扩展过程中具备一致的交互体验与可维护性。

\subsubsection{Spring Boot}
后端基于 Spring Boot 构建，采用约定优于配置的工程化模式，统一管理接口暴露、配置装配、依赖注入与业务分层。该框架能够快速搭建稳定服务骨架，降低样板代码量，便于后续模块化扩展与持续迭代。

\subsubsection{Spring Security}
系统使用 Spring Security 实现认证鉴权与访问控制，对不同角色和不同接口路径执行细粒度授权策略。结合令牌机制与安全过滤链，可有效支撑登录态校验、权限隔离和越权访问防护，满足版权业务的安全治理要求。

\subsubsection{MyBatis-Plus}
数据访问层采用 MyBatis-Plus，简化常见 CRUD、分页查询与条件构造开发，提升数据库操作效率与代码可读性。在保证 SQL 可控性的同时，减少重复开发工作，有利于业务快速落地与后续维护。

\subsubsection{MySQL}
关系型数据采用 MySQL 存储，主要承载用户主体信息、申请流程数据、审核结果、链上回执与审计日志等结构化数据。通过主外键约束、索引优化和事务机制，确保数据一致性、可追溯性与查询性能，满足版权确权业务对可靠存储与历史留痕的要求。

\subsubsection{MinIO}
系统使用 MinIO 存储原始软件包及相关附件，实现“链下保存原文”策略。对象存储路径与业务主键进行映射，支持文件快速定位、版本管理和后续审计回放。该方案可降低区块链存储成本，并在保证原始证据可复核的同时提升大文件处理能力。

\subsubsection{FISCO BCOS}
在确权环节，系统将关键摘要（如文件哈希、元数据哈希、证据根哈希）锚定至联盟链，实现“链上保存摘要、链下保存原文”的可信存证模式。通过链上交易回执、区块高度和时间戳，可形成不可篡改的证据记录，为版权归属核验、争议处理和司法取证提供技术支撑。

\section{总体设计}
\subsection{系统架构}
版权云链采用“前后端分离 + 分层服务 + 链上链下协同”的总体架构。前端以 Vue 3 提供业务界面与交互，后端以 Spring Boot 提供 REST API，并在服务层完成申请受理、风险评估、人工复核、上链存证和查询追溯等核心流程编排。数据层由 MySQL 持久化业务数据，MinIO 存储原始软件包，FISCO BCOS 联盟链负责关键证据摘要上链与交易回执固化。系统通过 JWT 与角色权限模型实现访问控制，通过审计字段和状态机机制保证流程可追踪、可回放、可核验。

	\caption{系统架构图}

\indent 系统采用分层架构，后端在统一权限与状态机约束下完成链下数据处理与链上确权存证。

\subsection{功能模块设计}
根据当前系统实现，平台功能可划分为六个核心模块，各模块通过统一接口规范与状态流转规则协同工作。

  \centering

\indent 模块之间通过申请编号、证据摘要、交易回执和角色权限进行联动，形成“申请-审核-上链-查询”闭环。

\subsubsection{用户与权限模块}
\indent\textbf{模块职责}：负责注册登录、主体识别、角色鉴权和会话管理。\par
\indent\textbf{业务逻辑}：用户完成注册或登录后，系统统一建立身份上下文，并按主体类型（个人/企业）与角色类型执行访问范围控制；后续业务请求在进入业务处理前均进行身份合法性与权限边界校验，确保不同角色仅能访问被授权的数据与操作。\par
\indent 该模块业务流程如图\ref{fig:user-auth-flow}所示。\par

\textbf{输入输出}：输入账号凭据与注册信息，输出令牌、角色、主体上下文。

\indent\textbf{业务逻辑}：系统将新申请与历史样本进行相似性比对，结合分值阈值和主体关系输出风险等级；低风险申请可直接进入后续确权流程，中高风险申请进入人工复核队列，由审核人员作出通过或驳回决策并触发状态迁移。\par
\indent 该模块业务流程如图\ref{fig:risk-review-flow}所示。\par

\indent 该模块业务流程如图\ref{fig:trace-query-flow}所示。\par

\subsection{版权申请提交与证据生成}
\indent\textbf{实现过程}：用户在申请页面上传软件压缩包并填写软件名称、描述等业务信息。服务端接收请求后按“先校验、再提取、后入库”的顺序执行，避免异常数据进入后续流程。具体步骤如下：\par
\indent 1）\textbf{上传完整性校验}：校验请求是否携带文件、文件大小是否在平台允许范围内、文件扩展名与 MIME 类型是否匹配，并对压缩包执行可解压测试（防止损坏包或伪造格式）。任一校验失败立即返回错误，不生成申请记录。\par
\indent 2）\textbf{内容一致性校验}：读取文件字节流后计算临时 SHA-256，并与上传过程中记录的字节长度、解压条目数量进行一致性比对，防止传输截断或重复拼接导致的“文件名相同、内容不同”问题。\par
\indent 3）\textbf{指纹提取与哈希生成}：系统同时生成三类摘要：文件哈希、规范化哈希、语义哈希。其中，文件哈希基于原始字节流直接计算 SHA-256，确保与原文件一一对应；规范化哈希通过对压缩包内文件按路径排序、统一换行符、剔除时间戳等非业务差异后再计算，用于降低构建环境差异带来的误报；语义哈希由代码特征（函数签名、关键调用片段、目录结构特征）提取后生成，用于相似性分析。\par
\indent 4）\textbf{业务数据落库}：先创建申请主记录（状态初始为待评估），再写入证据记录（含各类哈希与元数据），随后将原始文件写入对象存储并回填文件存储 ID，最终返回申请编号供前端追踪。\par
\indent\textbf{关键实现点}：采用“链下保存原文、链上保存摘要”策略；申请、证据、文件存储三类数据通过申请 ID 与文件存储 ID 关联形成完整证据链。文件哈希用于确权唯一性判定，规范化哈希与语义哈希用于后续相似性扫描，三者职责分离可同时兼顾“强一致校验”和“近似识别能力”。\par

\subsection{风险评估与人工复核}
\indent\textbf{实现过程}：系统在申请提交阶段同步执行相似性评估（即在写入申请记录前完成评分），整体流程为“候选样本加载 + 逐条评分 + 风险分级 + 状态流转”。\par
\indent 1）\textbf{候选样本加载}：系统从版权记录表中读取最近的 200 条且语义哈希不为空的历史记录（按创建时间倒序），作为本次比对候选集。\par
\indent 2）\textbf{maxScore 计算逻辑}：对每个候选记录执行一次语义哈希相似度计算，得到区间为 0--100 的整数分。单次评分公式为：
\[
\mathrm{score}=\mathrm{round}\left(\left(1-\frac{\mathrm{HammingDistance}(A,B)}{\mathrm{totalBits}}\right)\times100\right),
\]
其中，\(A\)、\(B\) 为两份语义哈希的字节序列，\(\mathrm{totalBits}=4\times\max(|hashA|,|hashB|)\)；若两哈希完全相同则直接记 100 分。系统最终取所有候选中的最大值作为 \texttt{maxScore}，并记录对应命中样本（matched）。\par
\indent 3）\textbf{风险分级与流转}：判定阈值由后端常量控制。\par
\indent \quad \texttt{HIGH\_RISK\_THRESHOLD=95}，\texttt{MEDIUM\_RISK\_THRESHOLD=85}。\par
\indent 判定规则为：\par
\indent \quad a. \(\texttt{maxScore} \ge 95\) 且跨主体（\texttt{sameSubject=false}）：风险级别 HIGH，进入人工复核（\texttt{PENDING\_REVIEW}）。\par
\indent \quad b. \(\texttt{maxScore} \ge 95\) 且同主体：风险级别 MEDIUM，视为同主体版本演进，不强制人工复核。\par
\indent \quad c. \(85 \le \texttt{maxScore} < 95\)：风险级别 MEDIUM，进入人工复核。\par
\indent \quad d. \(\texttt{maxScore} < 85\)：风险级别 LOW，自动通过风控并进入上链流程。\par
\indent 4）\textbf{状态机控制}：仅待复核状态允许审核员执行“通过/驳回”；系统在复核接口中再次校验 \texttt{PENDING\_REVIEW} 状态，防止重复审核和越状态处理。\par
\indent\textbf{关键实现点}：\texttt{maxScore} 采用“候选集合最大相似度”策略，解释性强且便于阈值治理；分级逻辑叠加“同主体修正”规则，可在拦截跨主体高相似风险的同时，减少同主体版本迭代的误判。\par

\subsection{测试目标与测试方法}
\indent\textbf{测试目标}：验证系统在“功能正确性、权限安全性、流程可追溯性、性能稳定性”四个方面满足上线要求，重点覆盖版权申请、相似性风控、人工复核、链上确权、结果查询与后台治理等核心链路。\par
\indent\textbf{测试方法}：采用“黑盒功能测试 + 接口联调测试 + 安全测试 + 性能压测 + 回归测试”的组合方式。\par
\indent 1）\textbf{黑盒功能测试}：依据业务流程设计正向与异常用例，验证输入、状态流转、输出结果与错误提示是否符合预期。\par

\indent 4）\textbf{性能压测}：使用 JMeter 对申请提交、状态查询、哈希查询等高频接口进行并发压测，观察吞吐量、响应时间与错误率。\par
\indent 5）\textbf{回归测试}：每次修复缺陷或调整阈值后，重新执行核心用例集，确保既有能力不回退。\par
